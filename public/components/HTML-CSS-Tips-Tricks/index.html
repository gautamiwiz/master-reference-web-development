Best Practises ~ Do's Dont's ~ Tips & Tricks

  • Always use semantic HTML
  • With component based approach, a lot of times the same CSS is being repeated over and over. Its best to think of the big picture to write DRY CSS.
  • Never use fixed width in terms of px, unless desperately required. Try giving widths in %. Do not give width 100% to all elements. This will cause overflow issues. Stick to width : auto which is default.
  • Try not using height as much unless required. Instead go with min-height. CSS works best when we donâ€™t define height and width at the same time
  • Use responsive units em and rem over px
  • Never use px for font sizing. It will cause issues to the users who prefer larger font size. + It causes issues in accessibility
  • When using css variables, always mention defaults "margin: var(--margin, 10px);""
  • Inline elements dont get vertical margin and padding.
  • Containing elements : Height/padding/margin given in % is based on its parent's width, not height!
  • The Box model
    • Border-box (includes border, padding and content width)
    • Does not include margin
    • Intrinsic and extrinsic sizing
  • Properties Inheritance
    • Only font properties are inherited.
    • Not everything is inherited
    • Not inherited : margin, padding, height, width, position
    • Not inherited :Button, input, optgroup, select, textarea
    • Solution to form elements inheritance. (add this to your css reset).
                  button,
            input,
            optgroup,
            select,
            textarea{
                font-family: inherit;
            }
    • Take advantage of Inheritance and write better CSS.
      • Example : In a banner if everything is centered, then the property 'text-align: center' must be give only once to the parent, instead of giving it to each child.
      • Fontsize fontfamily etch typography is inherited. That why if you want to write DRY code, just write the font size, font family, line height just once in the html tag.

  • Stacking context : any element that has position, opacity, transfor etc + many more.. creates a new stacking context. (this is related to the frustrating z-index.
    There are a lot of properties that create a new stacking context. When we create a new stacking context, it pulls that element onto that "new plane", putting it "above" the other elements on the page that are part of the normal flow.
    While they are pulled up, this does not mean you can use a z-index on them. So, for example, if you declare opacity: .5 on an element, and then try to give it a z-index, that z-index would have no effect. To use z-index, the element either needs a position other than static, or it must be a flex or grid child.
